# PostgreSQLデバッグ体験ハンズオン

PostgreSQL のソースコードをビルドし、意図的にバグを仕込んで `gdb` で解析するまでをハンズオンで体験してみよう、という記事です

# 1. 環境構築
以下のDocker環境を使います

https://github.com/Yutosaki/postgres-test

:::message
chezmoiというdotfilesの管理ツールを使用しており、ここは個人の設定ファイルに書き換えてもらって構いません
:::

docker 立ち上げ方
 ```bash
docker image build -t pg-test .
```

cloneしたdockerファイルのあるディレクトリ内で、上記コマンドを使用してイメージをビルド
その後、コンテナを作成する際に`--cap-add=SYS_PTRACE`と`--security-opt seccomp=unconfined`というオプションを付けます。
基本的にDockerでは、コンテナがホストのOSを乗っ取らないようにプロセスなどの機能に制限を掛けています。しかし今回は、デバッグをしたいのでそれらの機能を一部使える様にします
- SYS_PTRACE: プロセスが別プロセスのメモリを覗いたり一時停止させる機能を許可します
- seccomp=undefined: Dockerがデフォルトで禁止している、危険な可能性のあるシステムコールを全て許可するというものです。(`ptrace`など。これを使用して、悪意のあるプログラムが他のプロセスのパスワードを見たり、挙動を書き換えたりできてしまうため、デフォルトでは禁止されている)
```bash
docker container run -it --name pg-test --cap-add=SYS_PTRACE --security-opt seccomp=unconfined pg-test
```

## 2. ビルド・インストール手順
では、コンテナに入って作業を進めていきましょう

まずはpsotgresのソースコードをダウンロードします

```bash
git clone https://github.com/postgres/postgres
```

### ソースの構成（configure）

デバッグシンボルを有効にし、最適化を無効化した状態で `Makefile` を作成します。

```bash
./configure --prefix=$HOME/pgsql-debug --enable-debug --enable-cassert CFLAGS="-O0 -g"
```

### ビルド（並列実行）

`-j`オプションを付けることで指定したコア数で並列にmakeをしてくれます

https://qiita.com/ymdymd/items/312c9f554d4ffb1f8dc6

`nproc`でマシンの論理プロセッサ数を取得できます

```bash
make -j$(nproc)
```

以下のコマンドで、正しくmakeができたかテストできます
```bash
make check
```
### インストール

作成した実行ファイルなどを指定した場所に置くことができます
今回は`./configure`の時に指定した、$HOME/pgsql-debugに実行ファイルなどを起きます
```bash
make install
```

## 3. 環境変数設定

### 実行パスの追加

どこからでも自分の作成した実行ファイルが叩けるようにpathを通します
```bash
export PATH="$HOME/pgsql-debug/bin:$PATH"
```

### データディレクトリの指定

通常、postgreを起動するときはデータ(データの本体以外にもWALや設定ファイルなど全ての情報)をどこに保存するかを毎回指定しなければなりません。しかし、以下のPGDATAを設定しておくとpostgreの各コマンドが自動で判断してくれます。設定しない場合は`-D`オプションで`pg_ctl start -D /home/pgdata`などと毎回指定しなければなりません

```bash
export PGDATA="$HOME/pgdata"
```

### 反映確認

```bash
which postgres
```

---

## 4. DB の初期化と起動

### データベースの初期化

```bash
initdb -D $PGDATA
```

### サーバの起動

```bash
pg_ctl start -l logfile
```

### 動作確認（DB 作成と接続）

```bash
createdb testdb
psql testdb
```

---

## 5. GDB の使用方法

### バックエンドプロセスの PID を取得

```sql
SELECT pg_backend_pid();
```

### プロセスにアタッチ

```bash
gdb -p <PID>
```

---

## 補足

* `--enable-debug` と `-O0 -g` により、最適化を無効化しつつ詳細なデバッグ情報を付与しています。
* `--enable-cassert` により、アサーションチェックが有効化され、内部不整合の検出が容易になります。
* `gdb` ではブレークポイント設定（`break`）やスタックトレース確認（`bt`）が重要です。
